# Alertmanager Routing Runbook

## Purpose

This runbook documents how Prometheus alerts are routed to Alertmanager and then delivered to the local webhook receiver.

The goal of this phase is not production paging. The goal is to prove the alert delivery path:

Prometheus alert rule -> Alertmanager -> receiver -> evidence.

## Components

| Component | Purpose |
|---|---|
| Prometheus | Evaluates alert rules and sends firing/resolved alerts |
| Alertmanager | Groups, deduplicates, routes, and dispatches alerts |
| alert-webhook | Local test receiver that stores Alertmanager webhook payloads |
| docs/evidence/phase-14a/alertmanager-routing-test.md | Evidence generated by the test script |

## Local ports

| Service | URL |
|---|---|
| Prometheus | http://127.0.0.1:9090 |
| Alertmanager | http://127.0.0.1:9093 |
| Alert webhook receiver | http://127.0.0.1:9094 |
| LFCS Dashboard test port | http://127.0.0.1:3002 |

## Alertmanager configuration

Alertmanager config path:

    ops/alertmanager/alertmanager.yml

The local receiver is:

    local-webhook

The local webhook URL inside Docker Compose is:

    http://alert-webhook:8080/alertmanager

## Prometheus configuration

Prometheus sends alerts to:

    alertmanager:9093

This is configured in:

    ops/prometheus/prometheus.yml

## Manual verification

Check Prometheus sees Alertmanager:

    curl -fsS http://127.0.0.1:9090/api/v1/alertmanagers

Check Alertmanager readiness:

    curl -fsS http://127.0.0.1:9093/-/ready

Check webhook receiver health:

    curl -fsS http://127.0.0.1:9094/healthz

Check received webhook events:

    curl -fsS http://127.0.0.1:9094/events

## Controlled test

Run:

    npm run test:alertmanager-routing

The test stops the LFCS Dashboard container to make the existing LFCSDashboardDown alert fire.

The test then verifies that the alert reaches the local webhook receiver.

The test restarts the LFCS Dashboard container before exiting.

## Expected evidence

The evidence file should show:

- Compose services running
- Prometheus has an active Alertmanager target
- Alertmanager is ready
- webhook receiver is healthy
- LFCS Dashboard was stopped intentionally
- LFCSDashboardDown reached the webhook receiver
- LFCS Dashboard was restarted
- /readyz returned ready after recovery

## Operational interpretation

Prometheus alert rules alone are not enough.

A complete local alerting path requires:

1. an alert rule
2. Prometheus alertmanager target configuration
3. Alertmanager routing
4. a receiver
5. evidence that the receiver got the alert

## Production limitations

This local setup is not production paging.

Current limitations:

- webhook receiver is local only
- no Slack, email, PagerDuty, Opsgenie, or incident.io integration
- no authentication on the local webhook receiver
- no Alertmanager high availability
- no long-term notification archive
- no silence policy beyond manual Alertmanager UI usage

Future production-style improvements:

- route critical alerts to real notification channels
- use separate warning and critical receivers
- add silence and inhibition documentation
- add Alertmanager HA if running multiple Prometheus/Alertmanager instances
- add notification screenshots or incident tickets as evidence
